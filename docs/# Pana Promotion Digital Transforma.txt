## System Architecture: Detailed Blueprint

**High-Level Flow (Textual Diagram)**:

```
[Client-Side (Next.js 16 App Router)] ↔ [API Routes (SDK Wrappers + TanStack Hooks)] ↔ [ERP Backend (Frappe MVP Customizations)] ↔ [External: Fenanpay Webhooks]
  - Bidirectional: Order creation → TanStack mutation → SDK call → BOM decrement + Sonner toast
  - Validation Layer: Zod schemas on hooks; retry logic on transients (e.g., 3x webhook fails → manual override)

```

**Core Layers Breakdown**:

1. **Backend (VersaForge ERP - Customized Frappe MVP)**:
    - **Foundation**: Retain SDK singleton (`frappeClient` in `/lib/frappe-client.ts`) for all db/auth/call/file ops. Enhance `handleApiRequest` (in `/lib/api-template.ts`) with toast-friendly payloads: Success `{success: true, data: T, message: 'e.g., Stock synced'}`; Error `{success: false, error: 'e.g., Depletion risk', details: '...', statusCode: 500}`.
    - **Customization Depth (Regular Until Sprint 3)**: No bespoke doctypes; use inherent config (e.g., Item doctype for substrates/pigments). BOM hierarchies via `getDocList('BOM', {filters: {item: 'apparel-layer'}})`.
    - **Module Linkages**: Enforced via matrix—e.g., Inventory outputs to Manufacturing (threshold alerts trigger quote embeds).
    - **Data Ingress Protocol**: UTF-8 CSV/Excel (headerless); Frappe's `frappe.utils.csv_to_list` for parsing. Edge: Dupe excision via `getValue('Item', name, 'item_code')` uniqueness checks.
    - **Pitfall & Mitigation**: Over-reliance on mocks risks real-data variances (e.g., temporal mismatches in vendor logs)—counter: Sprint 4 segmented infusions (100-record batches) with `frappe.db.rollback` fallbacks.
2. **Frontend (Next.js 16 App Router - Modular Templates)**:
    - **Route Strategy**: Template-based for scalability—e.g., `[slug]/page.tsx` as base for products/blog; route groups `(marketing)` vs `(ecom)` for non-overlapping layouts. Global layout (`app/layout.tsx`): `<QueryClientProvider><ThemeProvider><Toaster /></QueryClientProvider>`.
    - **Themes**: TBD (dark/light via `next-themes` + CSS vars: `-bg: #0a0a0a; --accent: #00D4FF?`—probe: Neon for creative Pana audience, but test contrast ratios >4.5:1 via WAVE tool).
    - **Client-Side Refactor (TanStack Query v5+)**: Migrate all fetches/mutations to hooks (e.g., `useQuery({queryKey: ['items'], queryFn: () => frappeClient.db.getDocList('Item')})`). Optimistic updates for orders (e.g., `useMutation({onMutate: (newOrder) => queryClient.setQueryData(['orders'], old => [...old, newOrder]) })`). Invalidation: `queryClient.invalidateQueries({queryKey: ['stock']})` on webhook callbacks.
        - **Scope**: Client pages only (ERP dashboards like `/erp/inventory` + e-com like `/products`); ~25 components refactored (5/sprint).
        - **Pitfall**: Cache staleness in low-activity (e.g., >24h stock queries)—mit: `staleTime: 5 * 60 * 1000` (5min); probe: Does this align with Pana's real-time needs, or add `refetchInterval: 30000`?
3. **Client-Side Enhancements (Hooks, Validation, Notifications)**:
    - **Hooks Architecture (`/hooks` Dir)**: Composable, categorized for reuse—subdirs prevent bloat (e.g., 20+ hooks without collisions).
        - **/hooks/data**: TanStack wrappers (e.g., `useItemsQuery.ts`: `export const useItemsQuery = (filters?: Filters) => useQuery({queryKey: ['items', filters], ...})`).
        - **/hooks/domain**: Business logic + Zod (e.g., `useBOMValidation.ts`: `const schema = z.object({components: z.array(z.object({item: z.string()})).min(1)}); const {handleSubmit} = useForm({resolver: zodResolver(schema)})`).
        - **/hooks/ui**: Helpers (e.g., `useToastMutation.ts`: Wraps Sonner—`toast.success(data.message)` on success; `toast.error(error.details, {action: {label: 'Retry', onClick: mutate}})`).
        - **Patterns**: Composability (e.g., `useOrderFlow` = `useQuery(['order']) + useMutation(['update'])`); memoization (`useMemo` for deps); shared via `hooks/index.ts` barrels.
        - **Implementation Script**: (1) Scaffold dir Day 1 Sprint 1; (2) Generate 3-5 hooks/sprint via Grok prompt: "Composable TanStack hook for BOM decomposition, Zod-validated, Sonner-integrated"; (3) Test: Jest mocks for offline (e.g., `vi.mock('@/lib/frappe-client')`).
        - **Scalability Probe**: Handles 2x SKU growth? Stress: Simulate 500 queries in CodeSandbox—expect <200ms; fallback: Paginate with `useInfiniteQuery`.
    - **Validation (React Hook Form + Zod)**: Strict, TS-inferred schemas for all forms (e.g., item creation: `z.string().min(1).regex(/^[a-z0-9-]+$/)` for item_code). Uncontrolled inputs for perf; error display via `<ErrorMessage>{errors.item_code?.message}</ErrorMessage>`.
        - **Edge Cases**: BOM arrays (min 1 component); Fenanpay amounts (>0 ETB). Probe: Overly rigid for creative customizers—allow partial submits?
    - **Notifications (Sonner)**: Top-right, 3s auto-dismiss; variants (success: green, error: red with retry). Integrate SDK: `if (response.success) toast.success(response.message)`. Accessibility: ARIA labels; undo for mutations (e.g., "Order queued—undo?").
